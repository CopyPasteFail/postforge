Postforge Architecture

Activation, Delegation, and Output Surfaces

This page mirrors the Mermaid diagrams in docs/architecture.md, but gives them a cleaner browser presentation for repo walkthroughs, onboarding, and design review.

Simple scope A single compact pipeline for readers who just want the end-to-end story in plain language.
Swimlane scope The same end-to-end story, but organized by actor so delegation is obvious.
System scope Shows how the host agent activates the MCP server and where execution is delegated.
Tool scope Maps individual MCP tools to the orchestrator methods and downstream components they trigger.
Runtime scope Focuses on run stages, auth interruptions, polling, retry, skip, and final posting handoff.
Output scope Makes the persistent files and directories explicit.

Host vs server

The host agent still owns ranking, copywriting, and user-facing choices. The server owns tool execution, persistence, and browser automation.

Current runtime path

RunStage includes a few reserved or older values, but the diagrams above focus on the stages actively driven by OrchestratorAgent.

LinkedIn output nuance

output/linkedin/<runId>/ is provisioned with the run, but the active LinkedIn path mainly operates in the browser and opens the image folder for manual attachment.
